## Directory Purposes

**`.planning/codebase/`:**
- Purpose: GSD planning framework documents (generated by `/gsd-map-codebase`, consumed by `/gsd-plan-phase` and `/gsd-execute-phase`).
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md (if `quality` focus), TESTING.md, CONCERNS.md, STACK.md, INTEGRATIONS.md.

**`revisiting_Arch.md` (root):**
- Purpose: Master architecture and design document (self-contained, assumes zero prior context).
- Contains: System context, 19 core decisions (D1–D19) + 12 eval design decisions (E1–E12), authoritative data facts (measured against real Kambi catalog), representative queries (golden eval set), open questions, plan/next steps, constraints, glossary.
- **Read this first** before any other file — it is the authority for all downstream decisions.

**`resolver/`:**
- Purpose: LLM extraction stage (decision 11 + 19).
- **`schema.ts`:** Zod schema for the text-valued QueryPlan (decision 18). Encodes discriminated unions (`status`, `subject.kind`, `line.kind`), guarded objects (`odds`, `attrFilter`), the sport enum. This is what Haiku's structured output emits.
  - Key types: `Subject` (player | team | either_match_team | event), `Line` (numeric | binary), `Odds`, `AttrFilter`, `Selector`, `EventScope`, `QueryPlan`.
  - Constants: `BUILT_SPORTS` = generated at startup from `football/groups.json` (today `["FOOTBALL"]`).
  - Assumptions: Zod is a dependency (future `package.json`).
- **`extractor-prompt.md`:** The bounded extraction prompt (decision 16 + 19). Three-step procedure: (1) decide sport + status, (2) scope `event_scope` (teams, players+roles, competition, level, stage, time), (3) extract selectors with subject binding, market concepts, line/odds/attrFilter. Includes universal rules, boundaries, one worked example. Off-corpus (never reuses eval queries/entities/markets — decision 19 sub-rule).

**`eval/`:**
- Purpose: Golden eval harness design & seed data (decisions E1–E12, plan steps 1 + 3).
- **`gold-record.ts`:** Zod schema for gold records — mirrors `QueryPlan` from `resolver/schema.ts`, but every groundable leaf wrapped as `Grounded { id: number|number[], accept: string[] }`. Grounded id is the grading source of truth (E9); accept[] is diagnostic-only (triage + future text-fidelity layer).
  - Key types: `GoldSubject`, `GoldSelector`, `GoldEventScope`, `GoldRecord`.
  - Imports: `BEHAVIOR_TAG_IDS` from `behavior-tags.ts`.
- **`gold.seed.jsonl`:** Seed golden records, one per line. Each record: `{ id: "g001", query: "...", tags: [...], expect: { status, sport, event_scope, selectors }, notes: "..." }`. ~3 seed records exist (2 football, 1 tennis/no-sport); target ~50–70 behavior-tagged records (E7, ~5 per tag, including abstain cases).
  - Authoring rule (E8): human writes query+structure, fills ids via neutral name-search (not the grounder under test), second pass re-checks.
- **`gold.meta.json`:** Metadata stamp (E11): schema version, catalog version, counts (criterions, clubs, players), generatedFrom (data sources), validationNote.
- **`behavior-tags.ts`:** Behavior taxonomy (E7). Defines ~17 tags (binding, coref-his, coref-his-team, line-vs-price, line-no-number, attrFilter, player-role, level, stage, time, abstain, either-team, yes/no-line, odds-only-bounds, self-correction, age-normalize, sport-default); each tagged with tier (critical | soft) and description/example. Constants: `BEHAVIOR_TAG_IDS`, `BEHAVIOR_TAGS`, `CRITICAL_TAGS`, `SOFT_TAGS`.
- **`scorer.spec.md`:** Scoring specification (algorithm, not runnable code yet). Covers: inputs (gold + runs × 5), load/validation (E11), run protocol (E10: temp 0 ×5), scoring axes (status gate, sport, selector alignment by market id, binding, line/odds), verdict (strict pass on costly facets), aggregation per behavior tag, ship gate (critical = 100%, soft ≈90% aggregate, E12), failure triage (E4). Includes worked example (g001 with binding swap).

**`football/`:**
- Purpose: Kambi catalog data (per-sport partition, football only).
- **Core catalog data** (authoritative, measured against real Kambi datasets):
  - **`football_criterions.json`**: 607 criterion records (`{ id, sport, name, shownInLive, shownInPreMatch, categoryNames[], boTypeNames[] }`). The join hub of the market star (decision 4).
  - **`football_categories.json`**: 64 category records (`{ id, name, mappings: [{criterionId, boType, boTypeName}] }`). Maps (criterion, boType) pairs — per-sport grouping.
  - **`football_participants.json`**: 32,587 player + 1,784 club records (`{ id, kind, sport, name, clubId, competitionIds[], countryTeamId, ...}`). Semi-static entity data.
  - **`football_betoffertypes.json`**: ~28 betoffertype records (`{ id, label }`). Universal across sports (ids 1–128), not per-sport.
  - **`groups.json`**: Hierarchical group forest (competitions, confederations, nation groups, etc.). Root: universal node (id 1, sport NOT-SPECIFIED); direct children: sports (FOOTBALL, ESPORTS, OLYMPIC_GAMES, SPECIAL_BETS, NON_SPORT, NOT-SPECIFIED) — sports filtered from non-sports. FOOTBALL node is the root of the football competition subtree (used as sport enum generation source, decision 17).
- **Enrichment data** (static, hand-kept):
  - **`aliases.json`**: Deterministic, auditable hand-written aliases (e.g., `"SOT" → "Shots On Target"`). Used at grounding (decision 5).
  - **`derived-aliases.json`**: Auto-generated aliases from criterion name tokenization (FTS5 index source). Used for fallback / long-tail coverage.
- **Fixture/tournament data** (snapshot for testing):
  - **`2010133908_worldcup.json`**: World Cup 26 fixture snapshot. Used for end-to-end eval testing.
- **Refactor/tooling scripts** (Python):
  - **`refactor_participants.md`**: Notes on participant schema normalization.
  - **`refactor_participants.py`**: Normalize player/club records (e.g., ensure competitionIds array shape, strip nulls).
  - **`refactor_criterions.md`**: Notes on criterion structure (market mappings, category relationships).
  - **`merge_worldcup.py`**: Merge WC 26 fixture metadata into the catalog (e.g., add round/kickoff info).

## Naming Conventions

**Files:**
- TypeScript source: `*.ts` (e.g., `schema.ts`, `gold-record.ts`, `behavior-tags.ts`).
- Markdown docs: `*.md` (e.g., `extractor-prompt.md`, `scorer.spec.md`, `refactor_participants.md`).
- JSON data: `*.json` (catalog, eval, fixtures).
- Python scripts: `*.py` (refactoring, merging).
- JSONL (line-delimited JSON): `*.jsonl` (e.g., `gold.seed.jsonl` — one gold record per line).

**Directories:**
- Lower-case, underscore-separated (e.g., `resolver`, `football`, `eval`, `.planning`).

**Naming patterns:**
- Catalog files: `football_[entity].json` (e.g., `football_criterions.json`, `football_participants.json`).
- Eval files: `gold.*` (e.g., `gold.seed.jsonl`, `gold.meta.json`, `gold-record.ts`).
- Refactor notes: `refactor_[entity].md` (e.g., `refactor_participants.md`).
- Behavior tags: `behavior-tags.ts` (singular domain, plural tags).

## Where to Add New Code

**New TypeScript implementation (planned steps 2–7):**
- **Build pipeline:** `pipeline/build.ts` (TBD) — pull Kambi catalog, join enrichment, embed vectors, build SQLite.
- **Grounding:** `grounding/grounding.ts` (TBD) — market vectors+alias, entity trigram+alias, competition tree, attrFilter resolution.
- **Static store loader:** `store/loader.ts` (TBD) — load SQLite into RAM at boot, expose in-memory maps.
- **Executor:** `executor/executor.ts` (TBD) — run grounded plan, call live layer, fetch betoffers, apply filters.
- **Live event layer:** `live/live-layer.ts` (TBD) — fixtures, bracket, kickoff, lineup; separate service or same process.
- **Eval runner:** `eval/scorer.ts` (TBD) — implement `scorer.spec.md` (load gold, validate, run resolver ×5, score, aggregate per tag, check gate).

**Tests:**
- Unit tests co-located: `*.spec.ts` or `*.test.ts` alongside source (TBD, no test framework yet).
- Eval tests: `eval/scorer.ts` + `gold.seed.jsonl` (E10: 5× per query at temp 0).

**New catalog data:**
- **For a new sport:** Create `[sport]/[sport]_criterions.json`, `[sport]/[sport]_participants.json`, etc. Extend `football/groups.json` with the new sport as a top-level node. Update `BUILT_SPORTS` at runtime.
- **For enrichment:** `football/enrichment-region-table.json` (TBD, ~48 NT → confederation rows). Position + age feed: external (TBD).

**New gold records:**
- Add to `eval/gold.seed.jsonl`, one per line. Validate against `GoldRecord` (zod) + E8 authoring rules. Tag with behaviors (E7). Re-check ids on catalog rebuild (E11).

**Prompt refinements:**
- Edit `resolver/extractor-prompt.md` only on **universal reasoning** changes (binding, coref, line/odds, sport-inference). Sport facts / coverage → `gold.seed.jsonl` + `eval/behavior-tags.ts` (decision 16, bounded-prompt constraint). Off-corpus rule: never re-use an eval query/entity/market in the prompt example (decision 19 sub-rule).
