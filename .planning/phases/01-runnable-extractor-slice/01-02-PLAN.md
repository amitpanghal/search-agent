<objective>
Refine the extract slice with the two caching behaviors that make the harness cheap to iterate (Phase 2) and honest at release time: a response cache keyed by (query + prompt-contents-hash + model) so re-runs make zero API calls, and a `--fresh` flag that bypasses the cache and runs the query 5x at temperature 0 for a reproducibility check.

Purpose: The Phase 2 scorer will re-run the same corpus repeatedly; without a cache that is needless API spend. The `--fresh` 5x surfacing is the release-grade reproducibility signal (E10) - this phase only DISPLAYS the 5 results; the all-5-pass scoring is Phase 2.
Output: `resolver/cache.ts`; cache read/write wired into `resolver/cli.ts`; the `--fresh` flag with 5-block output + an "all 5 identical: yes/no" line; the gitignored `.cache/extract.json` created at runtime.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information disclosure | Cache file contents / key | mitigate | The cache stores only the query and the model's response plan - NEVER the API key (Task 1 forbids it). The key is read from env only. `.cache/` is gitignored (from Plan 01) so the cache is never committed. |
| T-02-02 | Information disclosure | Cache KEY derivation | mitigate | `cacheKey` hashes (query + prompt CONTENTS + model id) only - the API key is NOT an input, so the key cannot leak the secret (Task 1). |
| T-02-03 | Tampering | Cached value reused without re-validation | accept | Cached values are plans this process itself produced AND already passed `QueryPlan.safeParse` before being written; a local-only cache with no external writer. Re-parsing on read is optional and not required for this milestone. Low risk (single-user CLI, gitignored file). |
| T-02-04 | Information disclosure / Injection | NL query handling | mitigate | The query is only hashed into the cache key and sent as the Anthropic `user` message content; it is NEVER interpolated into a shell command or file path (the cache path is a fixed constant, not query-derived) (Task 1/Task 2). |
| T-02-05 | Tampering | Stale cache after prompt/model change | mitigate | Editing `resolver/extractor-prompt.md` or changing `MODEL` changes the sha256 key -> a fresh miss, so a stale response can never masquerade as current (Task 1, verified Task 3 step 4). |
</threat_model>
